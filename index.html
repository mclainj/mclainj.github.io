Galactic Fleet Admiral

This game is a turn-based strategy game that consists of allocating defensive and offensive ships to defend your planets and attack the enemy's planets, respectively. This project was implemented in a rudimentary programminng language called Jack. The game ran on a simple virtual computer I built throughout my computer organization course. Although the language it was implemeted in was limited, it did support some object-oriented principles, making OO design of the player/opponent classes possible. The AI opponent's logic was based on a decision tree, with a set of psuedo-random choices between multiple logical options to make the AI seem more competent.

Cave Flyer

Cave Flyer is the first game I developed in Unity. It is a fairly simple game that focuses on crafting a "skilled-operator" feel for the player as they progress through the game. This is accomplished through a tight control system, and increasingly unforgiving level design. To counteract the difficulty of some of the later levels, I added checkpoint and lives systems, so that the player can die three times before being sent back to the beginning level. This ended up being necessary for many of the people I had playtest the game, as the final level incorporates enemies that track and follow the player. While this game didn't contain advanced Unity topics, I learned much from this experience. It acted as a solid introduction to the Unity workflow.

Arbor Under Siege

When I decided to create a rails shooter game, I had to pay homage to the Star Wars Trilogy Arcade (1998) game that I played many times growing up. I used this as an opportunity to learn more about importing and maintaining assets in Unity. The ship models were free on turbosquid.com (however I borrowed the music from my pal, Mr. Williams). I also used this project to learn how to use the terrain tools and texturing in Unity. The movement of the player rig and enemy tie fighters are controlled through the timeline system.

I think the most satisfying aspect of the game is the flight controls, and establishing what kind of transformations the ship should experience with varying input and position on the screen. The player's movement is defined relative to the camera, with adjustable clamping. I found this fun to implement, as I was going through hierarchical transformations in my Computer Graphics class around this time.

I've found my greatest challenge in this game is the quantity of keyframe animations required to populate the world space, and make the player feel like they're in a hectic battle zone. I still have quite a bit more animation to go, and tidying of the UI before I am ready to call this game finished. I also plan to give the enemies blasters of their own, and a shield mechanic for the player to match it.

Minotaurus (Prototype)

Minotaurus is an ongoing turn-based game I'm developing based on the LEGO board game of the same name. I'm currently implementing this game to be a player vs. single AI opponent, however it will be expandable to multiple player/AI entities in the future. The goal is to get to the center goal of the maze before the opponent does, however the distance you can move is determined by the roll of a die. There are two twists to this game. The first of which is that the player and opponent can roll the "Minotaur" on the die, and control a neutral character for that turn that can send the opponent back to their start. The other mechanic that can be rolled on a die is the ability to activate or deactivate a gate on the map. This allows the players to engage in interesting scenarios, such as trying to block the opponent from getting to the goal, or blocking one's own path to stay safe from the Minotaur.

The current map and assets are placeholders while I develop the mechanics of the game. The blue sphere represents the player, red sphere represents the AI, and the green cube is the finish goal. So far, I have utilized the Nav Mesh system and some other heuristic methods to define the AI behavior. When a gate is activated along the AI's path, it will reroute to the next shortest path using a Nav Mesh Agent component. This is implemented using a variant of the A* pathfinding algorithm. Unfortunantely, the Nav Mesh system does not make it easy to access what objects are along a current path (I was getting inconsistent results), so I created a seperate method to calculate which gates to activiate/deactivate. This method uses two determining features: a raycast toward the finish goal, and a distance calculation of a gate with respect to the player and finish goal. The changing states of a gate are accounted for real-time in the Nav Mesh using the carving feature, which avoids the computationally expensive task of rebaking the entire mesh. This creates a mostly rational decision making process for the AI, similar to how humans might make the same calculation (save for the raycast). I may write my own Nav Mesh traversing algorithm to make the AI's decisions even more accurate in the future, however the smartest AI does not necessarily make for the most fun AI to play against.

The next aspect of Minotaurus I'll be working on is the "Minotaur" itself. It will be a neutral character on the map that only moves when either the player or AI rolls "Minotaur" on the die. It should be relatively simple to implement, as when the player rolls it, it will simply use the same player controller methods to move. When the AI rolls the minotaur, it will find the shortest path to the player using the same algoritm as the AI uses to get to the finish goal. When the Minotaur game object makes contact with the player or AI, it will reset the Minotaur and colliding entity to their starting positions. Following the implementation of the Minotaur, I plan to polish the game, and consider alterations to the mechanics that would allow for the player to traverse the corridors in first-person. Paired with a minimap, I believe this could foster some great gameplay moments.

Pancake Flipper

This is a group game project from my programming studio course. It was developed using the Agile methodology, with daily scrum meetings, and test driven development. The concept is that the player competes against an AI to "flip" randomly ordered pancakes until they are in order. This project was interesting because it was implemented in C++2a, and several iterations of graphics packages were used. This meant that the game engine code we wrote had to be modular enough to interact properly with ncurses (C++), FLTK (C++), and Tkinter (Python). The AI used a min-heap tree to store and traverse the possible moves it can make at the start of the game. While an algorithm such as A* would have been more efficient, we were required to use a variant of the minimax tree data structure. A difficulty system was implemented that determined how many "flips" ahead the AI could compute. The AI's logic worked as intended, making a tie the best outcome for the player at the highest difficulty.

What I found most valuable about this project was learning how to follow a development paradigm with a team. We also spent a considerable amount of time documenting our design, and created a nearly exhaustive interaction diagram before coding, which made the implementation quite smooth considering the scale.

Software Rasterizer

This was an interesting assignment from my ongoing computer graphics class. It is a C++ program that rasterizes OBJ files according to certain parameters. After reading in the vertex data from an OBJ file, the triangles are created using barycentric coordinates and converted to image space to be drawn. Z-buffering is implemented to represent depth in the image. Depending on the command line arguments, the program will either color the pixels according to Z-buffer depth, use random per-vertex coloring, or apply a gradient across the varying Y-position of the image. It then generates a PNG based on that color information. The image scales to whatever pixel width and height are passed as arguments.

GLSL Shaders

The following are the result of a few shader programs I wrote in GLSL, as part of a C++ OpenGL program. In order going down, the shading models used to create the three bunnies are Blinn-Phong, silhouette, and cel shading. The Blinn-Phong shader emulates the physics behind real world illumination quite closely. The silhouette shader simply checks if each vertex normal is orthogonal to the camera and colors that fragment black if it is. If not, it is colored white. It produces a fascinating "hand-drawn" looking effect. The shader on the final bunny (and the teapot at the top, and image at the bottom) is implemented using cel shading. This is a shading program that is somewhat of a combination of the previous two shaders, except it discretizes the colored fragments across 4-5 colors.
